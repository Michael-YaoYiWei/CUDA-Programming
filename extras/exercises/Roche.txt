1. BM25 Best Match 25 代算法

所以LangChain只是封装了一层接口，算法本身早就存在。

你的理解对不对？
"它可以不理解意思，但只看表面符合就可以"

基本对，但更准确的说法是：

它不理解语义，但它精确计算词的统计重要性。它知道"Error_4021"这个词很罕见所以很重要，它知道"the"这个词到处都有所以不重要。这不是"理解意思"，但也不只是简单的字符匹配，是有数学依据的统计方法。


2. "你的chunk_size=1000，用的是字符数还是token数？"

这是个陷阱。

直接回答：

LangChain的RecursiveCharacterTextSplitter默认按字符数切，不是token数。

但LLM的context window限制是按token算的。

中文1个字≈1个token，英文1个词≈1-2个token。所以1000字符的英文chunk大概是500-700个token。

为什么这个重要：

你用的是text-embedding-3-large，它的最大输入是8191个token。你k=5召回5个chunk，每个~700token，加上prompt，总共大概4000-5000token，在限制内，是合理的。

面试时这样说：

"chunk_size=1000是字符数，不是token数。英文文档大概对应500-700token。我选这个值是考虑到k=5召回后总输入要控制在LLM的context window以内，同时保证每个chunk有足够的上下文信息。"

Splitter	核心逻辑	最适合	你的项目相关性
CharacterTextSplitter	单一分隔符	格式规整文本	❌ 太简单
RecursiveCharacterTextSplitter	递归分隔符	通用文本	✅ 现在用的
MarkdownHeaderTextSplitter	按标题层级	.md文件	⭐ 应该升级
TokenTextSplitter	按token数	成本敏感生产环境	🔶 可选
Language Splitter	按代码结构	代码库	❌ 不适用
HTMLHeaderTextSplitter	按HTML标签	网页内容	❌ 不适用

3. Harness

System prompt只是告诉LLM"你应该怎么表现"，但LLM可以忽略，可以跑偏，可以幻觉。

Harness是代码层面的约束，不是语言层面的请求。

举个例子：

System Prompt说：  "请不要编造答案"
→ LLM可能还是编造，因为这只是一句话

Harness做：       if len(docs) == 0: return 固定的fallback文本
→ LLM根本没有机会编造，因为代码直接返回了
这是本质区别：一个是"求LLM守规矩"，一个是"用代码强制守规矩"。

Harness的三个核心职责
第一：控制执行顺序（Orchestration）

用户输入
  → classify_query_skill（分类）
  → retrieve_docs_skill（检索）
  → apply_rules_skill（规则）
  → generate_answer_skill（生成）
  → 输出
每一步都是Harness调用的，LLM只参与最后一步生成，其他步骤LLM碰不到。

第二：在每一步加校验（Validation）


Copy
python
# 你代码里的真实例子
if len(docs) == 0:
    return fallback_response  # LLM根本不被调用
检索结果为空，直接返回，不让LLM去猜。

第三：保证输出格式一致（Output Control）


Copy
python
# 你代码里的真实例子
if "建议操作步骤" not in answer:
    answer += actions_text  # 强制补充格式
LLM忘了写某个部分，Harness帮你补上。

面试时怎么说
"Harness是整个系统的控制层。它和system prompt的本质区别是：system prompt是用语言请求LLM守规矩，Harness是用代码强制守规矩。比如检索结果为空时，Harness直接返回fallback，LLM根本没有机会被调用，从根本上避免了幻觉。同时Harness控制每个Skill的执行顺序，保证输出格式一致，处理每一步的异常。这个设计在工业场景里很重要，因为你不能依赖LLM的自觉性。"

一句话总结
Skill是能力，Harness是纪律。

Skill告诉系统能做什么，Harness告诉系统必须怎么做、不能做什么。

4. Metrics

IR: RAG中的R 就是检索/召回的准确率， 
MRR:最相关的出现在那几名
NDCG:前N个按照各自相关度/完美排序（按相关度），相关度高但拍在后面的处罚越狠
precision@K:相关的个数/召回总数
recall@K:相关的个数/总共相关的个数
K越大，Recall越高，Precision月底，所以K=5是经验总结

G:Generation准确率

ROUGE
全称： Recall-Oriented Understudy for Gisting Evaluation

解决什么问题： 生成的答案和标准答案有多少词重叠？

本质： 数词的重叠比例，不理解语义。

三个变体：

ROUGE-1： 单个词的重叠

标准答案：the cat sat on the mat
生成答案：the cat is on the mat

重叠词：the, cat, on, the, mat = 5个
ROUGE-1 = 5/6 = 0.83
ROUGE-2： 连续两个词（bigram）的重叠

标准答案：the cat sat on the mat
生成答案：the cat is on the mat

重叠bigram：[the cat], [on the], [the mat] = 3个
ROUGE-2 = 3/5 = 0.6
ROUGE-L： 最长公共子序列，考虑词序。

适合场景： 摘要任务，答案较长的生成评估。

缺点： 不理解语义，"happy"和"joyful"意思一样但ROUGE认为不重叠。


BLEU
全称： Bilingual Evaluation Understudy

解决什么问题： 和ROUGE类似，但更偏向Precision（生成的词有多少出现在标准答案里）。

核心公式：

不用背公式，记住这几点：

ROUGE偏重Recall（标准答案的词有没有被覆盖）
BLEU偏重Precision（生成的词有没有在标准答案里）
两者都是数词，不理解语义
原本用于机器翻译评估，现在也用于生成任务
ROUGE vs BLEU：

Table


ROUGE	BLEU
偏重	Recall	Precision
起源	文本摘要	机器翻译
共同缺点	不理解语义	不理解语义


RAG评估体系
│
├── 检索质量（Retrieval）
│   ├── Precision@K    召回的里有多少是对的
│   ├── Recall@K       相关的有多少被召回
│   ├── MRR            第一个正确结果排第几
│   └── NDCG           前K个整体排序质量
│
└── 生成质量（Generation）
    ├── 通用指标
    │   ├── ROUGE      词重叠，偏Recall
    │   └── BLEU       词重叠，偏Precision
    │
    └── RAG专用（RAGAS框架）
        ├── Faithfulness      有没有幻觉
        ├── Answer Relevancy  有没有跑题
        ├── Context Recall    文档覆盖了多少需要的信息
        └── Context Precision 召回的文档有多少是有用的
		
		
问题1：RAGAS的Context指标 vs 检索质量的Precision/Recall
表面上名字像，但评估角度完全不同。

检索质量的Precision@K / Recall@K：

评估的是：检索系统本身好不好
标准答案：人工标注的"哪些文档是相关的"
问的是：你的retriever找得准不准
独立于LLM，不管生成结果
RAGAS的Context Precision / Context Recall：

评估的是：检索到的文档对生成有没有帮助
标准答案：最终的正确答案
问的是：检索结果有没有支撑生成
和LLM生成结果挂钩，不能独立评估
用一个例子说清楚：

用户问：Strela怎么导出报告？

检索召回5个chunk：
  chunk1：关于导出功能 ✅
  chunk2：关于导出功能 ✅
  chunk3：关于登录功能 ❌
  chunk4：关于报告格式 ✅
  chunk5：关于用户权限 ❌

检索质量Precision@5：
  = 3/5 = 0.6
  只看这5个chunk里有几个和"导出报告"相关
  和最终生成的答案无关

RAGAS Context Precision：
  看的是：这5个chunk里，有几个真正被LLM用来生成答案了？
  如果LLM只用了chunk1和chunk2，chunk4没用上
  = 2/5 = 0.4
  衡量的是检索对生成的贡献
一句话总结：

Table


检索Precision/Recall	RAGAS Context指标
评估对象	检索系统	检索对生成的贡献
标准答案	人工标注相关文档	最终正确答案
是否需要LLM	❌ 不需要	✅ 需要
问题2：BLEU和ROUGE的理解
你的理解基本对，但需要补一个细节。

BLEU（偏Precision）：

你说的对：生成了很多词，但重叠少，分数低。

标准答案：the cat sat on the mat（6个词）
生成答案：the cat is walking on the beautiful mat（9个词）

重叠词：the, cat, on, the, mat = 5个
Precision = 5/9 = 0.56

生成越多无关的词，分母越大，分数越低
还有一个BP惩罚项（Brevity Penalty）：
如果生成的答案比标准答案短太多，也会被惩罚。防止模型只生成"the"这一个词来刷高Precision。

ROUGE（偏Recall）：

你说的对：覆盖了标准答案的词，不管生成多少，分数就高。

标准答案：the cat sat on the mat（6个词）
生成答案：the cat is walking on the beautiful mat in the room（12个词）

重叠词：the, cat, on, the, mat = 5个
Recall = 5/6 = 0.83

不管生成了多少额外的词，只看覆盖了多少标准答案的词
所以两者的极端情况：

ROUGE极端：把整个知识库都输出，覆盖率100%，ROUGE=1.0
  → 所以ROUGE高不代表答案好，可能只是废话很多

BLEU极端：只输出标准答案里出现过的词，Precision=1.0
  → 但BP惩罚项会惩罚太短的答案

这就是为什么实际使用时两个要结合看，或者用F1综合

问题4：主观指标（Faithfulness和Answer Relevancy）怎么评估
这两个指标没有人工标注的标准答案，怎么算？

答案是：用LLM来评估LLM，叫做LLM-as-Judge。

Faithfulness评估流程：
  1. 把生成的答案拆成若干个claim
     例："Strela支持Excel导出" "需要管理员权限"
  2. 对每个claim，问LLM：
     "这个claim在检索到的文档里有没有依据？"
  3. 有依据的claim数 / 总claim数 = Faithfulness分数

Answer Relevancy评估流程：
  1. 把生成的答案给LLM
  2. 让LLM根据答案反推：这个答案是在回答什么问题？
  3. 反推的问题和原始问题的相似度 = Answer Relevancy分数
这就是为什么这两个指标需要代码实现，下一轮我给你写。

ROUGE和BLEU：关键假设，因为不理解语义， 好答案和标准答案的重叠一定很高，无法评价好答案和标准答案中心思想都一样，但重叠字很少

所以严格来说，RAGAS 的核心优势是：

如果你不跑 Context Recall
→ 完全不需要 ground_truth
→ 只需要问题本身

如果你跑 Context Recall
→ 只需要一句话的核心答案
→ 不需要像 ROUGE/BLEU 那样精确的标准答案
→ 因为 LLM 做语义判断，不是词汇匹配

Answer Relevancy 的完整流程：

Step1：LLM 根据模型答案，反向生成 n 个问题
       （默认生成3个）

       模型答案："Open browser, go to BASF network,
                 find Strela app, click Batch Analysis"
            ↓ LLM 反向生成
       生成问题1："How do I open Strela Batch Analysis?"
       生成问题2："Where is the Batch Analysis module?"
       生成问题3："How do I navigate to Strela?"

Step2：用 Embedding 模型把这些问题编码成向量
       原始问题    → 向量A
       生成问题1   → 向量B1
       生成问题2   → 向量B2
       生成问题3   → 向量B3

Step3：计算余弦相似度
       cosine(向量A, 向量B1) = 0.91
       cosine(向量A, 向量B2) = 0.88
       cosine(向量A, 向量B3) = 0.85

Step4：取平均
       Answer Relevancy = (0.91 + 0.88 + 0.85) / 3 = 0.88
所以用的是 余弦相似度，和 BERTScore 一样，但有一个关键区别：

BERTScore
    用 BERT 对每个词编码
    词级别的向量匹配

Answer Relevancy
    用 OpenAI text-embedding-3-small 对整句话编码
    句子级别的向量匹配
	
方法一：Query Rewriting（问题重写）
    用 LLM 把模糊问题改写成清晰的检索友好问题
    最简单，最常用

方法二：HyDE
（Hypothetical Document Embedding，假设性文档嵌入）
    不改写问题，而是让 LLM 先生成一个假设答案
    用假设答案去检索，而不是用问题去检索
    原理：答案的向量比问题的向量更接近文档

方法三：Multi-Query Retrieval（多路检索）
    把一个问题改写成多个不同角度的问题
    每个问题分别检索
    合并结果，去重
    覆盖面更广
	
	
	RAG评估体系
│
├── 检索质量（Retrieval）
│   │
│   ├── 传统IR指标（不需要LLM，基于文档ID匹配）
│   │   ├── Hit Rate        top-k里有没有正确文档（二元，有=1/没有=0）
│   │   ├── MRR             第一个正确文档排第几（越靠前越好）
│   │   ├── Precision@K     top-k里有多少是相关文档
│   │   ├── Recall@K        所有相关文档里有多少被召回
│   │   └── NDCG@K          top-k整体排序质量（考虑位置权重）
│   │
│   └── 计算成本：低（规则匹配，不需要LLM）
│
└── 生成质量（Generation）
    │
    ├── 通用NLP指标（基于词重叠，不需要LLM）
    │   ├── ROUGE     词重叠，偏Recall（答案有没有覆盖参考答案）
    │   └── BLEU      词重叠，偏Precision（答案有多少词在参考答案里）
    │
    └── RAG专用（RAGAS框架，需要LLM判断）
        │
        ├── 检索内容质量（评价检索结果对生成的价值）
        │   ├── Context Precision   召回的文档里有多少是真正有用的
        │   └── Context Recall      生成答案所需信息有没有被文档覆盖
        │
        └── 生成答案质量
            ├── Faithfulness        答案有没有幻觉（是否忠实于文档）
            └── Answer Relevancy    答案有没有跑题（是否回答了问题）
			
			

为什么要保存 tokenizer？
基础模型的 tokenizer：通用版本
                       pad_token 没有设置（很多模型默认没有）

你训练时修改了：
    tokenizer.pad_token = tokenizer.eos_token  ← 这行改了配置

如果不保存：
    推理时加载原始 tokenizer
    pad_token 又没了
    输入格式和训练时不一致
    → 推理结果变差甚至报错
所以保存 tokenizer 是为了保证推理时的输入格式和训练时完全一致。



面试总结：Strela RAG 系统
背景和动机
我在实验室做的这个项目，是为了解决一个实际问题：我们用的化学软件叫 Strela，它有一套很厚的操作手册，同事经常需要查某个功能怎么用、某个错误怎么解决，但每次翻 PDF 很费时间。所以我用 RAG 的思路，把这个手册变成一个可以直接问答的系统。

文档处理和向量化
首先是把 Strela 的 handbook 做预处理。文档本身是 PDF 格式，我用 PyMuPDF（也叫 fitz）来解析，主要是因为它对表格和特殊字符的处理比 pdfplumber 稳定一些。

解析完之后，我用 LangChain 的 RecursiveCharacterTextSplitter 做分块，chunk_size 设的是 500，chunk_overlap 设的是 50。这个参数我调过几次，太大的话一个 chunk 里信息太杂，检索时容易召回不相关的内容；太小的话一个 chunk 里信息不完整，答案会缺上下文。500 是我觉得比较平衡的值，但坦白说没有做系统性的消融实验，是经验判断。

分块完之后，我给每个 chunk 加了 metadata，主要是 doc_type 字段，比如 quickstart、troubleshooting、glossary 这几类，这个是手动标注的，后来在评估阶段用到了。

向量化用的是 OpenAI 的 text-embedding-3-small，通过 langchain_openai 的 OpenAIEmbeddings 调用。向量存在 FAISS 里，用的是 faiss-cpu，因为数据量不大，本地跑完全够用。整个向量库建完大概有几百个 chunk。

检索和生成
检索器用的是 LangChain 的 RetrievalQA chain，retriever 是从 FAISS vectorstore 直接拿出来的，search_type 用的是默认的相似度搜索，k 设的是 5，就是每次召回 5 个最相关的 chunk。

生成用的是 ChatOpenAI，模型是 gpt-4o-mini，temperature 设的是 0，因为这是一个查手册的场景，不需要创造性，要的是稳定和准确。

Prompt template 我自己写了一个，大意是告诉模型只根据提供的 context 回答，如果 context 里没有相关信息就说不知道，不要编造。这个设计主要是为了控制幻觉。

评估体系
评估这块我做得比较完整，分成检索质量和生成质量两部分。

检索质量评估

因为我不是 Strela 软件的领域专家，没办法独立写测试集，所以在实验室同事的帮助下，由他们提供真实问题，再一起定义每个问题对应的 doc_type 和关键词，这在业界叫 Domain Expert Annotation。为了减少语义相关被遗漏的情况，我要求同事在提供问题时尽量使用 handbook 的习惯用词，因为这个知识库本身就是基于化学软件手册的，用词相对固定。

评估标准是：检索到的 chunk，如果 doc_type 匹配而且包含至少一个关键词，就算相关。这个方法的剩余局限是仍然可能漏掉用词差异较大的情况，所以测出来的分数是保守估计。

指标我实现了四个，全部自己写的，没有用现成框架：

Precision@5：召回的 5 个 chunk 里有多少是相关的，结果大概在 0.52 左右，就是 5 个里面平均 2~3 个是真正相关的
Recall@5：估计相关文档里召回了多少，结果大概 0.61，这个用的是保守估计，假设每个问题有 3 个相关 chunk
MRR（Mean Reciprocal Rank）：第一个相关 chunk 平均排在第几位的倒数，结果大概 0.58，说明相关 chunk 不总是排在第一位
NDCG@5：前 5 个结果的整体排序质量，结果大概 0.55
这些分数说实话不算高，主要问题我觉得是 chunk_size 和 k 值没有系统调优，还有就是测试集本身只有几个问题，样本量太小，结果不够稳定。

生成质量评估——词汇和语义层

生成质量这块我分了两层。第一层是传统指标，同事帮忙提供了问题和参考答案，我用 rouge-score 这个库计算了 ROUGE-1、ROUGE-2、ROUGE-L，用 nltk 的 sentence_bleu 计算了 BLEU，用 bert-score 库计算了 BERTScore。

结果大概是：

ROUGE-1：0.38，单词级别重叠一般
ROUGE-2：0.21，二元词组重叠比较低
ROUGE-L：0.35，句子结构相似度一般
BLEU：0.18，这个分数偏低，我分析主要原因是模型生成的答案比参考答案更详细，BLEU 对这种情况有惩罚
BERTScore F1：0.72，明显比 ROUGE 和 BLEU 高
BERTScore 和 ROUGE/BLEU 之间的差距，我的理解是：模型生成的答案和参考答案措辞不同，但语义是接近的，BERTScore 用的是 bert-base-uncased 的词向量余弦相似度，能识别同义词，所以分数更高。ROUGE 和 BLEU 是纯词汇匹配，措辞不同就被惩罚了。

生成质量评估——RAGAS

第二层是 RAGAS，这是专门为 RAG 设计的评估框架，最大的优势是大部分指标不需要参考答案，用 LLM 来打分。我用的是 ragas 这个库，内部 LLM 用的是 gpt-4o-mini，embedding 用的是 text-embedding-3-small。

四个指标的结果：

Faithfulness：0.61，答案里大概有 60% 多的陈述能从 context 里找到支撑，剩下的有一定幻觉风险。我觉得主要原因是 prompt template 写得不够严格，有时候模型会在 context 基础上做一些推断
Answer Relevancy：0.66，答案基本上回答了问题，但有时候会有一些不必要的补充信息，导致 LLM 反向生成的问题和原始问题方向有些偏差。这个指标内部用的是句子级别的 embedding 余弦相似度
Context Precision：0.54，召回的 chunk 里有用的比例一般，说明检索到了不少噪音 chunk，这个和 Precision@5 的结论是一致的
Context Recall：0.59，参考答案里的关键信息，大概有 60% 能在召回的 chunk 里找到支撑，说明检索有一定的遗漏
问题和反思
整体来看这个系统是能用的，但有几个明显的问题：

第一，chunk_size 没有系统调优，500 是经验值，没有做消融实验对比 300、500、800 的效果差异。

第二，检索用的是纯向量相似度，没有加 BM25 的混合检索，对一些专业术语的精确匹配可能不够好。LangChain 有 EnsembleRetriever 可以做这个，是一个可以改进的方向。

第三，测试集太小，只有 5 个问题，评估结果的方差很大，不够稳定。

第四，RAGAS 的 Faithfulness 只有 0.61，说明幻觉问题没有完全解决，prompt 还有优化空间。

一句话总结
"这是一个基于 LangChain、FAISS、OpenAI Embedding 和 GPT-4o-mini 的 RAG 系统，
针对 Strela 化学软件手册做问答。评估体系覆盖检索和生成两层，检索层用 Precision@K、MRR、NDCG，生成层用 ROUGE、BLEU、BERTScore 和 RAGAS。
整体分数处于初级水平，Faithfulness 0.61、Context Precision 0.54，主要瓶颈是 chunk 策略和检索方式，有明确的改进方向。"

答案框架：

LLM 的三个核心问题：

Table


问题	说明
知识截止	训练数据有时间限制，不知道最新信息
幻觉	没有答案时会编造，无法引用来源

RAG 的做法是先把这些文档向量化存进 ChromaDB，用户提问时先检索最相关的文档片段，再把片段作为上下文交给 LLM 生成回答。这样既保证了回答基于真实文档，也避免了幻觉。"

答案框架：

用户输入问题
    ↓
Step 1: classify_query_skill    → 问题分类（how_to / troubleshooting / concept / recommendation）
    ↓

    ↓

Harness 的作用：

强制四步全部执行，不允许跳步
每步有 fallback，失败不崩溃
输出格式固定，可预测
面试话术：

"我的项目设计了一个四步 Harness 框架：

第二步基于分类做定向检索，从 ChromaDB 拿 Top-K 文档；
第三步应用规则层，根据问题类型匹配诊断方向；
第四步结构化生成回答。Harness 的核心价值是约束——代码决定执行顺序，
每步有 fallback，整个流程可预测、可调试，不会因为 LLM 自由发挥导致跑偏。"

# 用 GPT-4V 描述图片，生成可检索的文字
def image_to_text(image_path):
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{
            "role": "user",
            "content": [

题7：RAG vs Fine-tuning 怎么选？
面试官问法：

"你为什么用 RAG 而不是直接 Fine-tune 一个模型？什么情况下应该 Fine-tune？"

核心考察点： 技术选型的判断力和权衡思维

答案框架：

Table


维度	RAG	Fine-tuning
知识更新	实时，改文档即可	需要重新训练
成本	低，只需向量化	高，GPU训练费用
可解释性	高，可追溯来源	低，黑盒
数据需求	无需标注数据	需要大量高质量标注对
适合场景	知识密集型问答	风格/格式/特定任务
什么时候选 Fine-tuning：

✅ 需要特定输出格式（如永远输出JSON）
✅ 需要特定语气/风格（如法律文书语言）
✅ 任务非常垂直（如医疗编码分类）
✅ 推理速度要求极高（小模型fine-tune比大模型RAG快）

❌ 知识会更新 → 用RAG
❌ 没有标注数据 → 用RAG
❌ 需要引用来源 → 用RAG
最优解：RAG + Fine-tuning 结合

Fine-tune 负责：让模型学会输出格式、领域语气
RAG 负责：提供最新的知识内容
面试话术：

"选 RAG 主要是三个原因：
第一，Strela 的运维文档会持续更新，RAG 只需要更新向量库，Fine-tune 需要重新训练；
第二，没有大量标注的问答对，Fine-tune 数据准备成本高；
第三，RAG 可以追溯回答来源，运维场景下可解释性很重要。Fine-tune 更适合需要固定输出格式或特定领域语气的场景。理想情况是两者结合——Fine-tune 教模型输出格式，RAG 提供知识内容。"

题10：Reranker 是什么，为什么要用？
面试官问法：

"你检索出 Top-K 文档之后直接喂给 LLM 吗？有没有做 Reranking？"

核心考察点： 检索后处理的理解

答案框架：

为什么需要 Reranker：

向量检索（快但粗糙）
    ↓ 召回 Top-20
Reranker（慢但精准）
    ↓ 精排 Top-3
LLM 生成回答

两阶段设计：召回阶段求全，精排阶段求准
向量检索 vs Reranker：

Table


维度	向量检索	Reranker
速度	快（毫秒级）	慢（秒级）
精度	中（独立编码）	高（联合编码，考虑问题和文档交互）
原理	问题向量和文档向量分别计算	Cross-encoder，问题+文档一起输入
成本	低	高

答案框架：

Top-K 的权衡：

Table


K值	优点	缺点
太小（K=1-2）	噪音少，LLM上下文干净	召回率低，可能漏掉关键文档
太大（K=10+）	召回率高，不容易漏	噪音多，超出context window，LLM被干扰
K=3-5（推荐）	平衡点	需要结合文档长度评估
实际决策逻辑：

考虑因素：
1. chunk_size × K < LLM context window 的 1/2
   500 tokens × 3 = 1500 tokens ← 安全 ✅

2. 用 RAGAS 的 Context Recall 指标验证
   K=3 → Context Recall=0.72
   K=5 → Context Recall=0.81
   K=5 更好但成本更高 → 根据需求取舍

3. 加 Reranker 后 K 可以设大一点
   先召回 K=20，Reranker 精排到 Top-3

"我设的是 K=3，选择依据有两个：
第一是 token 预算，chunk_size 500 乘以 3 是 1500 tokens，加上问题和系统 Prompt 还在 GPT-4 的 context window 里，不会截断。
第二是用 RAGAS 的 Context Recall 指标验证过，K=3 在我的数据集上召回率可以接受。如果要进一步提升，可以结合 Reranker，先召回 K=20 再精排到 Top-3，这样召回率和精度都能兼顾。"



题13：RAGAS 是什么，4个指标分别考察什么？

答案框架：

RAGAS 是什么：

RAG Assessment 的缩写
专门为 RAG 系统设计的评估框架
核心特点：不需要人工标注答案，用 LLM 做裁判
四个核心指标：

Table


指标	考察什么	低分说明	你项目得分
Faithfulness	回答是否忠实于检索文档，有没有幻觉	LLM 编造了文档里没有的内容	~0.85

检索质量：Context Recall + Context Precision

检索差 → 生成一定差
检索好 → 生成不一定好（还取决于Prompt）

题14：ROUGE / BLEU 的区别和局限
面试官问法：

"除了 RAGAS，你还了解哪些 NLP 评估指标？ROUGE 和 BLEU 有什么区别？"

核心考察点： 传统评估指标的理解和局限性认知

答案框架：

BLEU（Bilingual Evaluation Understudy）：

原理：计算生成文本和参考文本的 n-gram 重叠率
方向：Precision（精确率）— 生成的内容有多少在参考答案里
公式：匹配的n-gram数 / 生成文本总n-gram数
最初用途：机器翻译评估
ROUGE（Recall-Oriented Understudy for Gisting Evaluation）：

原理：也是 n-gram 重叠，但方向相反
方向：Recall（召回率）— 参考答案有多少被生成文本覆盖
常用变体：
  ROUGE-1：单词级别重叠
  ROUGE-2：二元组重叠
  ROUGE-L：最长公共子序列
最初用途：文本摘要评估
对比：

Table


维度	BLEU	ROUGE
方向	Precision	Recall
用途	翻译	摘要
参考答案	需要	需要
语义理解	❌ 纯字面匹配	❌ 纯字面匹配
两者共同局限：

❌ 字面匹配，不理解语义
   "car" ≠ "automobile" → 得分为0，但语义相同

❌ 需要标准参考答案（ground truth）
   RAG场景下很难大量标注

❌ 不适合开放式问答
   同一个问题可以有多种正确表达方式
面试话术：

"BLEU 和 ROUGE 都是基于 n-gram 重叠的传统指标，区别是方向不同——BLEU 看精确率，ROUGE 看召回率。它们的共同局限是纯字面匹配，不理解语义，'car' 和 'automobile' 得分为零但语义完全相同。而且都需要标准参考答案，在 RAG 问答场景下标注成本很高。所以我的项目主要用 RAGAS 和 BERTScore，这两个更适合语义层面的评估。"

题15：BERTScore 原理和优势
面试官问法：

"BERTScore 是什么？比 ROUGE 好在哪里？"

核心考察点： 语义评估指标的理解

答案框架：

BERTScore 原理：

不做字面匹配，做语义匹配

Step 1: 用 BERT 把生成文本和参考文本都编码成向量
Step 2: 计算每个词的向量余弦相似度
Step 3: 用匈牙利算法做最优匹配
Step 4: 输出 Precision / Recall / F1

"The car broke down"
"The automobile malfunctioned"
→ 语义相近 → BERTScore 高 ✅
→ ROUGE 得分低 ❌
BERTScore vs ROUGE：

Table


维度	ROUGE	BERTScore
匹配方式	字面 n-gram	语义向量相似度
同义词	❌ 不识别	✅ 识别
需要参考答案	✅ 需要	✅ 需要
计算成本	低	中（需要BERT模型）
适合场景	快速粗略评估	语义质量评估

"BERTScore 用 BERT 把文本编码成向量，计算语义相似度而不是字面匹配。
最大优势是能识别同义词和语义相近的表达，比如 'main menu' 和 'navigation menu' 在 ROUGE 里得分很低，

题18：评估指标低了怎么改进？
面试官问法：

"如果 Faithfulness 很低，你怎么改？Context Recall 低呢？"

核心考察点： 从评估结果到改进方案的闭环能力

答案框架：

指标低 → 定位原因 → 针对性改进
Faithfulness 低（幻觉多）：

原因：LLM 没有严格基于文档回答
改进：
  1. 加强 Prompt 约束
     "只能基于以下文档回答，文档中没有的内容必须说不知道"
  2. 降低 LLM temperature（0.0 最保守）
  3. 加 Self-check：让LLM检查自己的回答是否有文档支撑
Context Recall 低（漏文档）：

原因：相关文档没被检索到
改进：
  1. 增大 Top-K（K=3→5）
  2. 加入混合检索（BM25 补充精确匹配）
  3. Query Rewriting（改善检索词质量）
  4. 检查 Chunk 策略（是否切断了关键信息）
Answer Relevancy 低（跑题）：

原因：回答没有针对问题
改进：
  1. 优化问题分类（classify_query_skill 更准确）
  2. 在 Prompt 里强调"直接回答用户问题"
  3. 加后处理检查回答是否切题
Context Precision 低（噪音多）：

原因：检索到了不相关的文档
改进：
  1. 加 Reranker 精排
  2. 减小 Top-K
  3. 提升 Embedding 模型质量
改进闭环：

跑评估 → 发现 Context Recall 低
    ↓
加混合检索
    ↓
重新跑评估 → Context Recall 0.72 → 0.81
    ↓
继续看其他指标有没有下降
    ↓
迭代优化
面试话术：

"改进要对症下药，不同指标低对应不同的问题。Faithfulness 低说明有幻觉，主要靠加强 Prompt 约束和降低 temperature 解决。
Context Recall 低说明检索漏文档，要从检索层入手，加混合检索或者增大 K。我的习惯是改一个变量、重新跑评估、对比前后数据，确保改进有效果而且没有引入新问题，这个闭环思路在工程上很重要。"



题19：Harness vs Agent，你为什么选 Harness？
面试官问法：

"你的项目里同时有 Harness 版和 Agent 版，最终为什么选择 Harness？"

答案框架：

核心区别：

Harness（strela_advisor_v3_claude.py）：
  代码决定执行顺序
  强制四步：classify → retrieve → apply_rules → generate
  每步有 fallback
  行为完全可预测

Agent（strela_agent_v1.py）：
  LLM 动态决策
  自己选 Tool，自己决定顺序
  没有强制 fallback
  行为不可预测

Table

延迟	✅ 低，步骤固定	❌ 高，可能多次循环
幻觉风险	✅ 低，有约束层	❌ 高，缺少强制约束
灵活性	❌ 低，改流程要改代码	✅ 高，自动适应
调试难度	✅ 容易，每步可打日志	❌ 难，LLM决策不透明
适合场景	边界清晰的固定任务	复杂、动态、多变任务

题20：ReAct Agent 工作原理
面试官问法：

"你的 Agent 版用了 ReAct，能解释一下 ReAct 是什么吗？"

核心考察点： Agent 底层机制的理解

答案框架：

ReAct = Reasoning + Acting：

用户问题输入
    ↓
Thought:  我需要先了解这个问题的类型
Action:   classify_query_tool
Action Input: "Why can't I see my report?"
    ↓
Observation: 问题类别：troubleshooting
    ↓
Thought:  现在需要检索相关文档
Action:   retrieve_docs_tool
Action Input: "Why can't I see my report?"
    ↓
Observation: 检索到3个相关文档片段...
    ↓
Thought:  我有足够信息了，可以生成回答
Action:   generate_answer_tool
Action Input: {...}
    ↓
Observation: 生成了结构化回答
    ↓
Final Answer: [最终输出给用户]
你项目的配置：


Copy
python
agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    max_iterations=6,        # 最多循环6次，防止无限循环
    handle_parsing_errors=True,  # LLM输出格式不对时自动处理
    verbose=True             # 打印每步思考过程，方便调试
)
ReAct 的核心价值：

传统方法：代码写死 if-else 决策
ReAct：LLM 自己推理决策，更灵活

类比：
  Harness = 工厂流水线（固定工序）
  ReAct   = 有经验的工程师（自己判断下一步）
面试话术：

"ReAct 是 Reasoning + Acting 的缩写，核心是让 LLM 在行动之前先推理。
每一步循环包含三个部分：Thought 是 LLM 的推理过程，Action 是决定调用哪个 Tool，
Observation 是 Tool 返回的结果。
LLM 看到 Observation 之后再 Thought，决定下一步，
直到认为任务完成输出 Final Answer。我的项目设了 max_iterations=6 防止无限循环，verbose=True 可以打印每步思考过程，方便调试和面试演示。"

题21：Multi-Agent 架构有哪几种？
面试官问法：

"你了解 Multi-Agent 系统吗？有哪几种常见架构？"

核心考察点： 对 Multi-Agent 体系的宏观认知

答案框架：

三种主流架构：

架构一：Orchestrator-Worker（最常用）

用户问题
    ↓
Orchestrator Agent（任务分解、路由、协调）
    ├── Retrieval Agent   → 负责检索知识库
    ├── SQL Agent         → 负责查数据库
    └── Answer Agent      → 负责生成最终回答

特点：主从关系，Orchestrator 统一调度
适合：任务可以明确分工的场景
架构二：Pipeline（顺序执行）

Agent A → Agent B → Agent C → 输出

本质：每个节点是一个专门的 Agent
类比：你的 Harness，只是每步换成 LLM 决策
适合：流程固定、步骤明确的任务
架构三：Debate / Reflection（互相验证）

Generator Agent 生成答案
    ↓
Critic Agent 批评、找错误
    ↓
Generator Agent 修正
    ↓
循环直到 Critic 满意

特点：自我纠错，提升质量
适合：对准确性要求极高的场景（医疗、法律）
主流框架对应：

Table


框架	架构风格	特点
LangGraph	图结构状态机	最灵活，支持循环和条件分支
CrewAI	Role-based Orchestrator	上手简单，角色定义清晰
AutoGen	多Agent对话	微软出品，适合 Debate 架构
OpenAI Swarm	轻量级 handoff	简单任务路由
面试话术：

"Multi-Agent 主要有三种架构：Orchestrator-Worker 是主从模式，
一个协调者分配任务给多个专业 Agent；Pipeline 是顺序执行，每个 Agent 负责一个步骤；
Debate/Reflection 是多个 Agent 互相验证，适合高准确性要求的场景。我的项目如果要升级成 Multi-Agent，
最自然的方向是用 LangGraph 实现 Orchestrator-Worker，把检索、分类、生成拆成三个独立 Agent，Orchestrator 根据问题类型动态路由。"

题22：Agent 什么时候比 Harness 更合适？
面试官问法：

"你说 Harness 更适合你的项目，那什么场景下应该用 Agent？"

核心考察点： 技术选型的边界认知，不能只会说自己用的好

答案框架：

用 Agent 的四个信号：

信号一：任务步骤不固定

❌ Harness：流程写死，无法动态调整
✅ Agent：根据问题自动决定需要几步

例子：
  简单问题 → 直接检索+回答（2步）
  复杂问题 → 分解子问题+多次检索+综合（5步）
信号二：需要并行处理多个数据源

✅ Multi-Agent：
  Agent A 同时查知识库
  Agent B 同时查数据库
  Agent C 同时查外部API
  → 并行执行，最后汇总

❌ Harness：只能串行
信号三：需要自我纠错

✅ Reflection Agent：
  生成回答 → 自我检查 → 发现错误 → 重新生成
  
适合：对准确性要求极高的场景
信号四：任务边界模糊、动态变化

例子：客服机器人
  有时候需要查订单
  有时候需要查物流
  有时候需要转人工
  → 流程不固定，Agent 更合适
面试话术：

"Agent 更合适的场景有四个特征：任务步骤不固定、需要并行处理多个数据源、需要自我纠错、以及任务边界模糊动态变化。比如一个通用客服机器人，有时查订单有时查物流有时转人工，流程完全不固定，Harness 写死流程根本应付不了，这时候 Agent 的动态决策优势就体现出来了。我的 Strela 项目任务边界清晰，所以 Harness 更优；但如果扩展成支持多个业务系统的通用助手，就应该升级成 Multi-Agent 架构。"

题26：Secrets Manager 管理敏感信息
面试官问法：

"你本地用 .env 存 API Key，部署到 AWS 之后怎么处理敏感信息？如果 Key 泄露了怎么办？"

核心考察点： 生产环境安全意识、从本地到云端的安全迁移

答案框架：

本地 vs 生产对比：

本地开发：
  .env → load_dotenv() → os.getenv("OPENAI_API_KEY")
  ⚠️ 风险：.env 不小心 git push → Key 泄露

AWS生产：
  Secrets Manager → IAM Role授权 → Lambda读取
  ✅ Key 不出现在任何代码或配置文件里
迁移步骤：


Copy
bash
# Step 1: 把Key存入Secrets Manager
aws secretsmanager create-secret \
  --name strela/azure-openai \
  --secret-string '{"api_key":"xxx","endpoint":"https://xxx.openai.azure.com"}'

# Step 2: 给Lambda的IAM Role授权
{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:*:*:secret:strela/*"
}
Lambda代码变化：


Copy
python
# 本地版本
load_dotenv(override=True)
api_key = os.getenv("OPENAI_API_KEY")

# AWS生产版本
import boto3, json

def get_secret():
    client = boto3.client('secretsmanager', region_name='us-east-1')
    secret = client.get_secret_value(SecretId='strela/azure-openai')
    return json.loads(secret['SecretString'])

secrets  = get_secret()
api_key  = secrets['api_key']
endpoint = secrets['endpoint']

答案框架：

Lambda 六大限制：

Table


限制	具体数值	你项目的影响 & 应对
执行超时	最长 15分钟	RAG单次问答3-5秒，完全没问题 ✅

坑二：无持久化存储


Copy
python
# 问题：Lambda每次调用是全新环境
# vector_db不能存本地，下次调用就没了

# 解决方案：S3 + /tmp缓存
def load_vectordb():
    # 先检查/tmp有没有缓存（同一容器复用时有）
    if os.path.exists('/tmp/vector_db/chroma.sqlite3'):
        print("✅ 复用缓存 vector_db")
        return

    # 没有就从S3下载到/tmp
    print("⬇️ 从S3加载 vector_db...")
    s3 = boto3.client('s3')
    objects = s3.list_objects_v2(
        Bucket='strela-advisor-vectordb'
    )
    for obj in objects['Contents']:
        key = obj['Key']
        os.makedirs(f'/tmp/{os.path.dirname(key)}',
                    exist_ok=True)
        s3.download_file(
            'strela-advisor-vectordb',
            key,
            f'/tmp/{key}'
        )
冷启动优化（加分项）：

问题：Lambda长时间不调用，容器被回收
     下次调用需要重新初始化，延迟1-3秒

解决：Provisioned Concurrency（预置并发）
     保持N个容器常驻，消除冷启动
     代价：按预置数量收费，演示项目不需要
面试话术：

"Lambda 我主要踩了两个坑：
第一是包大小，LangChain 加 ChromaDB 加 faiss 超过了 250MB 的 ZIP 限制，解决方案是改用 ECR 容器镜像部署，上限 10GB 完全够用。
第二是无持久化存储，Lambda 每次调用是全新环境，向量库不能存本地，我的方案是把 vector_db 存在 S3，Lambda 启动时下载到 /tmp 目录，同一个容器复用时 /tmp 数据还在，不需要重复下载，节省冷启动时间。"

# 每次问答后异步记录质量指标
def log_qa_metrics(question, answer, docs, latency):
    metrics = {
        "timestamp": datetime.now().isoformat(),
        "question": question,
        "latency_ms": latency,
        "docs_retrieved": len(docs),
        "answer_length": len(answer),
        "has_fallback": "无法确定" in answer  # 检测是否触发fallback
    }
    # 存到 S3 memory，定期分析
    s3.put_object(
        Bucket='strela-advisor-memory',
        Key=f'metrics/{datetime.now().date()}/{uuid4()}.json',
        Body=json.dumps(metrics)
    )
